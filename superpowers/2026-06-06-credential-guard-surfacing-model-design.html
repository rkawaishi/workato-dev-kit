Credential Guard — 表出モデルへの再設計

Bash スキャンを「プログラム許可リスト（default-deny）」から「内容が LLM コンテキストに表出するときだけブロック（default-allow）」へ。もぐら叩きの偽陽性を構造的に解消する。

2026-06-06 block-credential-read.sh ×2 test_block_credential_read.py supersedes PR #176

1背景と問題

credential guard の Bash スキャンは SAFE_PROGS = {"workato"} ＋ git staging の許可リストにないプログラムが credential 名を含んだら全ブロックする default-deny だった。これは構造的に偽陽性（正当作業のブロック）を生み続ける。

巻き込まれていた正当コマンド

bundle exec workato exec -s settings.yaml.enc -k master.key # 暗号化設定でのローカルテスト（標準形） cp settings.yaml.enc bak · ./deploy.sh --settings settings.yaml.enc · curl --key client.key # 内容を表に出さないのに名前を含むだけでブロック

新しい正当ツールが出るたびに allowlist 追加が必要な「もぐら叩き」。コミット履歴がそれを物語る：全ブロック → Grep/Glob漏れ修正 → #175 ダンプ限定化 → #176 bundle exec展開。

2脅威モデル（確定版）

“ 防ぎたいのは credential の内容が LLM のコンテキストに入ること。bash 等で直接プログラムに入力・引数として渡す分は許容する。 ”

  • エージェントが credential を「取得して低レイヤーで再利用（迂回）」できるのは、まず内容が自分のコンテキストに入った時だけ。コンテキスト流入を断てば迂回の原資も断てる。
  • これは「サンドボックス」ではなく事故 / 迂回ガード。意図的な多段回避・シェル関数再定義・コネクタコードの secret print はスコープ外。
  • ネットワーク exfil もユーザーが「プログラムに流すのは OK」と明示的にスコープ外とした。

3設計判断 — 採用は案B「表出モデル」

不採用案A — default-deny / allowlist 維持

セキュア寄りだが偽陽性が残り続ける。脅威モデルが「内容のコンテキスト流入だけ」を問題視するため過剰。

採用 ✓案B — 表出モデル / default-allow

credential 名を含むセグメントは内容を stdout/stderr に表出させるときだけブロック。それ以外は許可。偏陰性リスク（未知プリンタ漏れ）は「表出ブロックリスト＋既知プリンタ明示」で十分 — 一次防御の Read/Grep/Glob は default-deny のまま、Bash はセカンダリ防御という多層構成で受ける。

不採用案C — Bash スキャン廃止

cat master.key 等を素通りさせ脅威モデルに反する。

4Bash 表出判定ルール

credential パターンを含むセグメントについて、表出条件に該当したら block / どれにも該当しなければ allow（default-allow）。実プログラムは既存のランナー展開（VAR= スキップ・bundle exec・env・command・time・nice の剥がし）で解決し、#176 の展開ロジックをこの解決器へ転用する。

BLOCK — 内容が表に出る

① Emitter プログラム

中身を吐く: cat tac nl head tail less more bat strings xxd od hexdump base64 grep rg sed awk cut sort uniq tr jq yq tee …

② 既知の平文プリンタ

openssl … -in <cred> / gpg -d / ヘルパー sdk decrypt / トークンに decrypt を含む呼び出し

③ シェル構文での吸い出し

$(…) / バッククォート本体に credential / emitter への入力リダイレクト < <cred>

④ インタプリタの読み出し

python|ruby|node|perl|php の -c/-e スニペット内に credential 名 → 保守的に block

⑤ git 専用オラクル

既存 git_segment_safe() を流用。「git-safe でない = 表出する」。show rev:path・diff・-p・-c alias=!cat・--no-index を捕捉

ALLOW — 表に出ない（プログラムに渡すだけ）

公式ツールでの暗号化設定操作

bundle exec workato exec -s settings.yaml.enc -k master.keyworkato edit settings.yaml.enc

staging / ファイル操作

git add settings.yaml.enccp · mv settings.yaml.enc bak

自作スクリプト・ツールへの引数渡し

./deploy.sh --settings settings.yaml.enccurl --key client.key https://…

出力ファイル名の偽陽性回避

workato generate schema --output=out.key

SAFE_PROGS 許可リストは廃止 — workato を特別扱いする必要がなくなる（emitter でないので自然に通る）。

5コンポーネントとデータフロー

INPUT (JSON) → tool_name 判定
├─
Read / Edit / Write / NotebookEdit → path_hit() → deny / allow
├─
Grep / Glob → path_hit() + 到達性 walk → deny / allow
└─
Bash → segment 分割 → contains_cred? → surfaces(seg)? → deny / allow
└ resolve_program()（ランナー展開）＋ emitter / known-printer / subst / interpreter / git-oracle
不変（一次防御 / default-deny） 刷新（二次防御 / 表出モデル）

エラーハンドリングは現行維持 — malformed JSON・パターンファイル欠如・Python 例外は fail-open（allow）。bash ラッパは exit 2 のみブロック。

6テスト方針

維持（BLOCK のまま）

  • cat master.key
  • grep token settings.yaml
  • openssl pkey -in id_rsa.key
  • python3 -c "open('master.key').read()"
  • git show HEAD:…settings.yaml
  • git add -p · git status -v · --no-index
  • git -c alias.x='!cat …'
  • sdk decrypt · comment-spoof
  • git status && cat master.key（連鎖）
  • bundle exec cat master.key（展開後 emitter）

新規追加（B が許可 = ALLOW）

  • bundle exec workato exec … -s settings.yaml.enc -k master.key
  • cp …/settings.yaml.enc …/settings.yaml.enc.bak
  • mv master.key connectors/x/master.key
  • ./deploy.sh --settings …/settings.yaml.enc
  • curl --key …/client.key https://example.test
  • env cat master.key → BLOCK（回帰ガード）
  • time cat master.key → BLOCK（回帰ガード）

allowlist 前提だった test_*_allows_workato_edit_enc 等は「emitter でないので自然に allow」へ意味を更新。両フック（Claude フル / Codex Bash-only）に適用。

8スコープ外（明示）

⚠ この再設計が守らないもの

  • 意図的多段回避（非credential名へコピー後に読む、シェル関数再定義、コネクタコードが secret を print）
  • ネットワーク exfil（プログラムへ流す行為は許容）
  • 一次防御（Read / Grep / Glob パスフック）の挙動変更は行わない
