# workato-dev-kit (framework development)

This repository is where the framework itself is developed — the skills, rules, docs, and scripts that ship to Workato users.
**This is not where Workato recipes are written**. Recipe development happens in an organization's workspace repository that adds this repository as a `kit/` submodule.

End-user-facing skills (like `/create-recipe`) and Workato-targeted rules live under [framework/claude/](../framework/claude/). The `.claude/` directory at the repository root holds **only the configuration needed to develop the kit itself** (kept separate to avoid mixing).

## Repository structure

```
workato-dev-kit/                  ← this repository
├── .claude/                      ← kit development (this directory; does NOT contain Workato rules/skills)
│   ├── CLAUDE.md                 # this file
│   └── settings.json             # minimum permission settings for kit development
├── framework/                    ← distribution (symlinked into user repositories)
│   ├── claude/                   # for Claude Code (canonical source)
│   │   ├── CLAUDE.md             # user-facing Workato development rules
│   │   ├── rules/                # Workato development rules (recipe format, etc.)
│   │   ├── skills/               # Workato development skills (/create-recipe, etc.)
│   │   ├── hooks/                # Workato hooks
│   │   └── settings.json         # template for the user's .claude/settings.json
│   ├── cursor/                   # for Cursor (generated by scripts/sync_agents.py)
│   │   ├── rules/                # *.mdc (workato-project.mdc is hand-written)
│   │   ├── skills/               # SKILL.md
│   │   └── hooks.json            # hand-written
│   ├── codex/                    # for Codex CLI (generated)
│   │   └── skills/               # SKILL.md (slash syntax `/foo` rewritten to `$foo`)
│   ├── gemini/                   # for Gemini CLI (generated)
│   │   └── skills/               # SKILL.md
│   └── AGENTS.md                 # agent-neutral rules (generated; aggregated CLAUDE.md + rules). GEMINI.md is the same artifact
├── docs/                         # knowledge base (connector info, logic, platform)
├── guides/                       # lifecycle and other guides
├── connectors/                   # custom connector doc templates (users add connector.rb here)
├── projects/                     # patterns/ for user projects, etc.
├── scripts/
│   ├── sync_agents.py            # syncs framework/claude/ → framework/{cursor,codex,gemini}/ + AGENTS.md
│   ├── sync-cursor-rules.sh      # backwards-compat wrapper (just invokes sync_agents.py)
│   └── workato-api.py
├── templates/                    # templates for user repositories (gitignore, etc.)
├── setup.sh                      # runs in the user repository and creates symlinks
└── README.md
```

## User view

In a user repository (one that has added the kit as a submodule), running `bash kit/setup.sh` creates the following. Cursor receives real file copies; everything else is a symlink:

| Path in user repository | Distribution | Link / copy source |
|---|---|---|
| `.claude/rules/<file>.md` | symlink | `kit/framework/claude/rules/<file>.md` |
| `.claude/skills/<name>/` | symlink | `kit/framework/claude/skills/<name>/` |
| `.claude/hooks/<file>.sh` | symlink | `kit/framework/claude/hooks/<file>.sh` |
| `.claude/settings.json` | initial copy | hook paths expanded to `kit/framework/claude/hooks/...` |
| `.claude/CLAUDE.md` | initial copy | content imported from `framework/claude/CLAUDE.md` |
| `.cursor/rules/<file>.mdc` | **copy** | `kit/framework/cursor/rules/<file>.mdc` |
| `.cursor/skills/<name>/...` | **copy (recursive)** | `kit/framework/cursor/skills/<name>/...` |
| `.cursor/hooks.json` | **copy** | `kit/framework/cursor/hooks.json` |
| `.cursor/.kit-manifest` | generated | list of kit-managed files (used for prune decisions on the next setup) |
| `.agents/skills/<name>/` | symlink | `kit/framework/codex/skills/<name>/` |
| `.gemini/skills/<name>/` | symlink | `kit/framework/gemini/skills/<name>/` |
| `AGENTS.md` | symlink | `kit/framework/AGENTS.md` |
| `GEMINI.md` | symlink | `kit/framework/AGENTS.md` (same artifact) |

Users **can add their own files** in the same directories as these symlinks/copies. `setup.sh` preserves non-symlink files on the symlink side, and on the Cursor side it preserves files not present in the manifest.

**Why Cursor uses copy only**: Cursor IDE does not reliably resolve symlinks for `.cursor/rules/*.mdc` or `.cursor/skills/<name>/` (many bug reports on forum.cursor.com — silent load failures, files not detected after a restart, etc.). v2.5 partially fixed it but regressions have been reported. We switched to real-file copies; when the kit is updated, users re-run `bash kit/setup.sh` to re-copy.

### Organization knowledge layer `org/` (only exists on the user side)

Users cannot write to `docs/` (symlinked into the kit), so corrections and organization-specific information that overrides the kit's docs go in **`org/docs/`** (mirroring the same hierarchy as `docs/`). Learning skills like `/learn-recipe` write into `org/docs/...`. `setup.sh` never touches `org/`; the workspace repository commits it. See [framework/claude/rules/org-knowledge-overlay.md](../framework/claude/rules/org-knowledge-overlay.md) for details.

In this repository (the kit itself), `docs/` is the kit canonical knowledge — sync-style skills like `/sync-connectors` write to it. `org/` does not exist in the kit itself (only in user repositories).

## Development rules

### Editing the distribution (framework/)

- The canonical source is `framework/claude/`. Edit skills and rules here:
  - Skills: `framework/claude/skills/<name>/SKILL.md`
  - Rules: `framework/claude/rules/<name>.md`
- After editing, always run **`python3 scripts/sync_agents.py`**. This regenerates `framework/{cursor,codex,gemini}/` and `framework/AGENTS.md`. Include both the `framework/claude/` edits and the regenerated distribution artifacts in the same commit (CI enforces sync correctness).
- References like `@.claude/rules/...` in skill prose are written **from the user view** (not `framework/claude/rules/...`). For Cursor, the sync step rewrites them to `.cursor/rules/...mdc`.

### Cursor-only files (hand-written)

The following live in `framework/cursor/` and are NOT generated; edit them directly:

- `framework/cursor/rules/workato-project.mdc` — `alwaysApply: true` project-wide rule
- `framework/cursor/hooks.json` — Cursor hook configuration

Everything else (8 rules + skill set) is overwritten by sync — do not edit it directly.

### Changing setup.sh

- When changing the user-side directory structure (`.claude/`, `.cursor/`, `docs/`, `guides/`, `scripts/`, `templates/`), maintain **idempotency**: users will re-run `bash kit/setup.sh` after framework updates.
- Run with existing user repositories in mind. Avoid destructive changes (anything that deletes user files beyond re-linking existing symlinks).

### Editing docs/

- `docs/connectors/<provider>.md` and similar are knowledge bases that users reference via `@docs/...`. Keep link targets working from the user view (do not rewrite them to kit-internal paths).

## Testing

- After changing `setup.sh`, run it in a test directory and confirm the symlinks/copies are created correctly:
  ```
  TEST=/tmp/kit-test-$$ && mkdir -p "$TEST" && cd "$TEST"
  ln -s "$(git -C <kit-repo> rev-parse --show-toplevel)" kit
  bash kit/setup.sh
  ls -la .claude/rules .claude/skills .claude/hooks      # should be symlinks
  ls -la .cursor/rules .cursor/skills .cursor/hooks.json # should be real files (copies)
  cat .cursor/.kit-manifest                              # list of kit-managed files
  # Confirm idempotency on a second run (no diff)
  bash kit/setup.sh
  ```
- After changing `scripts/sync_agents.py`, run it in the repository and confirm `framework/cursor/rules/*.mdc` and `framework/cursor/skills/*/SKILL.md` are regenerated. CI (`.github/workflows/sync-check.yml`) catches drift automatically.

## Notes

- Workato-targeted skills like `/create-recipe` are **not used** in this repository. To prevent accidental invocation, the root `.claude/skills/` is empty.
- Per-developer local settings go in `.claude/settings.local.json` (not git-managed). Accumulate allow-list entries there.
