# /learn-pattern

Workato に精通したエキスパートが、完成レシピから汎用的な構築パターンを抽出し `docs/patterns/recipe-patterns/` に蓄積するスキル。
パターンの判別はエキスパート（利用者）が行い、このスキルはそれを効率的にドキュメント化する。

## 使い方

- `/learn-pattern <file-path>` — 特定のレシピを分析
- `/learn-pattern <project-name>` — プロジェクト内の全レシピを分析
- `/learn-pattern` — 引数なしで対話開始

## 手順

### 1. 対象レシピの読み込みと構造サマリー

指定された `.recipe.json` を読み込み、構造を要約して提示する:

- トリガー種別（provider + name）
- 全ステップの一覧（provider, action, ネスト構造）
- 使用している構造要素: ループ（`foreach`）、分岐（`if/else`）、エラーハンドリング（`try`）、Recipe Function 呼び出し、ブロッキングアクション等

```
## レシピ構造サマリー

**トリガー**: <provider> / <name>
**ステップ数**: N

0. [trigger] <provider>/<name>
1. [action] <provider>/<name> — <label>
2. [loop] foreach
   2.1 [action] <provider>/<name>
   2.2 [if] <condition>
       2.2.1 [action] ...
3. ...

**構造要素**: ループ, 分岐, エラーハンドリング
```

### 2. エキスパートにパターンの判断を聞く

構造サマリーを提示した上で、利用者にどのパターンを見出したかを聞く:

```
このレシピからどのような構築パターンを抽出しますか？

例:
- 「ページネーションループで全件取得するパターン」
- 「承認待ちのブロッキングと Slack ボタン連携のパターン」
- 「エラー時にリトライ後 Slack 通知するパターン」

既存パターン一覧: <_index.md のパターン名を列挙>
既存パターンへの追記・更新も可能です。
```

### 3. 既存パターンの確認

`docs/patterns/recipe-patterns/_index.md` を読み、利用者が指定したパターンが既存かどうか確認する。

- **既存パターンに該当**: パターンファイルを読み、レシピから得られた新しい知見を提示して追記内容を相談
- **新規パターン**: テンプレートに従って新規作成

### 4. パターンの記述

利用者の判断と対象レシピの具体的な構造をもとに、パターンをドキュメント化する。

#### 新規パターンのテンプレート

```markdown
# <パターン名> (<English Name>)

## いつ使うか

| 条件 | 該当 |
|---|---|
| <条件1> | Yes |
| <条件2> | Optional |

## レシピ構成図

\```
[トリガー/コンテキスト]
    │
    ├── [Action/Loop/IF] <step description>
    └── ...
\```

## ステップ構成

| # | Provider | Action | 目的 |
|---|---|---|---|
| N | provider | action_name | 説明 |

## 設計判断ポイント

| 判断 | 推奨 | 理由 |
|---|---|---|

## 既知の注意点

- <注意点>

## 参照

- <関連ドキュメントやパターン>
```

**記述のガイドライン**:

- パターンは **特定のコネクタや組織に依存しない汎用的な形** で記述する
- レシピの具体的な値（チャンネル名、プロジェクト名等）は抽象化する
- 「なぜこの構造にするのか」の理由を設計判断ポイントに含める
- エキスパートが口頭で伝えるような実装上の勘所を「既知の注意点」に記録する

#### 既存パターンへの追記

追記前に Grep で重複がないか確認し、新しい知見のみ追記する:
- 新しい設計判断ポイント
- 新しい注意点（Gotcha）
- 構成図のバリエーション
- 参照実装の追加

### 5. インデックスの更新

新しいパターンを作成した場合、`docs/patterns/recipe-patterns/_index.md` のパターン一覧テーブルに行を追加する。

### 6. 確認

作成・更新したパターンファイルの内容を利用者に提示し、過不足がないか確認する。
エキスパートの知見が正確に反映されているかが最も重要。

## 汎用パターンの判断基準

このリポジトリに蓄積するのは **Workato プラットフォームに紐づく汎用パターン** のみ。
判断は利用者が行うが、迷った場合の目安:

| 蓄積する（汎用） | 蓄積しない（組織固有） |
|---|---|
| ページネーションループで API の全件を取得する構造 | 「freee API のページネーション仕様」 |
| 承認 → 分岐 → 後続処理の構造 | 「IT Onboarding で Jira の特定フィールドを使う」 |
| try/catch → リトライ → 通知の構造 | 「#it-helpdesk チャンネルにエラーを通知」 |
| ブロッキングアクションの配置ルール | 特定プロジェクトのレシピ分割方法 |

組織固有の適用事例はプロジェクトの `DESIGN.md` に記録する。

## 出力

完了後、以下を報告:

- 分析したレシピ
- 作成・更新したパターンファイルと内容のサマリー
- 次に `/learn-recipe` でフィールド知識も蓄積することを案内
