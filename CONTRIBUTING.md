# Contributing to workato-dev-kit

Thanks for considering a contribution to workato-dev-kit.
This document covers development setup, the PR flow, and how to add new skills.

Japanese contributors are welcome to discuss in either language on issues — the maintainers respond in English or Japanese as appropriate.

All new documentation contributions should follow the canonical wording listed in [i18n/GLOSSARY.md](i18n/GLOSSARY.md).

---

## Development setup

This repository can be developed **on its own** — you do not need to add it as a `submodule` of a consumer workspace.

```bash
git clone https://github.com/rkawaishi/workato-dev-kit.git
cd workato-dev-kit
```

Requirements:

- `bash` (zsh users on macOS still need `bash`)
- `python3` (3.8+, standard library only)
- `git`

The Workato CLI is not required for kit development itself (only on consumer workspaces).

See [.claude/CLAUDE.md](.claude/CLAUDE.md) for the detailed repository layout and directory roles.

## Editing rules for distributables (`framework/`)

**The canonical source is `framework/claude/`.** Edit skills, rules, and hooks here.

```
framework/
├── claude/        ← canonical source (edit here)
├── cursor/        ← generated by scripts/sync_agents.py (do not edit directly)
├── codex/         ← same
├── gemini/        ← same
└── AGENTS.md      ← same
```

> **Important**: after editing `framework/claude/`, always run `python3 scripts/sync_agents.py`
> to regenerate Cursor / Codex / Gemini / AGENTS.md.
> CI ([sync-check workflow](.github/workflows/sync-check.yml)) detects drift automatically and blocks the PR.

Cursor-specific files maintained by hand (not regenerated by sync):

- `framework/cursor/rules/workato-project.mdc` — project-wide rule with `alwaysApply: true`
- `framework/cursor/hooks.json` — Cursor hook configuration

Other Cursor files are overwritten by sync, so do not edit them directly.

## Branch naming

Use the form `<type>/<slug>`. `<type>` follows Conventional Commits:

| Type | Use for |
|---|---|
| `fix/` | Bug fixes |
| `feat/` | New features |
| `docs/` | Documentation-only changes |
| `chore/` | Build, tooling, miscellaneous |
| `refactor/` | Refactors without behavior changes |
| `ci/` | CI configuration |
| `test/` | Test additions or fixes |

Examples: `fix/setup-python-shell-escape`, `docs/governance-files`, `feat/add-pull-recipe-skill`.

---

## Commit messages

Follow [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/).

```
<type>(<scope>): <subject>

<body — why the change is needed>

Closes #<issue-number>
```

Example:

```
fix(setup): pass paths to python via env to avoid shell-escape breakage

setup.sh's `python3 -c` interpolated shell variables directly, which
broke when paths contained quotes. Switched to a heredoc + os.environ
pattern to isolate the inputs.

Closes #92
```

## Pull request flow

1. **Find or open an issue**: open one first if no existing issue applies (small typo fixes are fine without).
2. **Create a branch**: `git checkout -b <type>/<slug>`
3. **Implement + verify locally**:
   - If you edited `framework/claude/`, run `python3 scripts/sync_agents.py`.
   - If the area has tests, run `python3 -m unittest scripts/tests/test_*.py`.
4. **Commit**: Conventional Commits style.
5. **Push + open PR**: `gh pr create --base main` or via the GitHub Web UI.
6. **CI check**: the `sync-check` workflow must be green.
7. **Review**: maintainers handle the merge.

PR bodies should include a "Summary", a "Test plan", and `Closes #N` where applicable.

## Tests

Current tests:

When you add a feature, please add tests where reasonable.

## New skill checklist

When adding `framework/claude/skills/<new-skill>/SKILL.md`:

- [ ] The `description` frontmatter is written so a model can mechanically decide when to invoke the skill
- [ ] `allowed-tools` declares every tool the skill needs (including external ones like Chrome MCP)
- [ ] `@.claude/...` references in the skill body are written from the consumer view (the post-symlink relative path)
- [ ] No responsibility overlap with another skill
- [ ] `python3 scripts/sync_agents.py` has been run and `framework/{cursor,codex,gemini}/skills/<new-skill>/` has been generated
- [ ] Any impact on `framework/AGENTS.md` has been considered
- [ ] [guides/skills-reference.md](guides/skills-reference.md) and the README's skill table list the new skill

## Editing the knowledge base (`docs/`)

The knowledge base under `docs/` (connector docs, logic, platform features) is written so users can reference it via `@docs/<path>`.
Do not rewrite links to kit-internal paths.

Organization-specific corrections and additions belong in `org/docs/` on the consumer side. The kit's `docs/` should only contain Workato-derived general information.
